# Permissions

The current foundation only answers safe Discord liveness messages: `/ping`, DM `ping`, and guild mention `@Gigi ping`. It does not execute privileged Discord actions yet, so no capability checks are live beyond configuration and database readiness.

The next Discord slices should keep the earlier authority rule: surface is presentation, not authority. DM, slash commands, message commands, buttons, and plugin triggers must resolve the same guild identity and capability model before doing work.

## Planned Capability Shape

| Capability | What it allows |
| --- | --- |
| `plugin.install` | Install or configure guild plugins |
| `plugin.run.<id>` | Use a plugin-provided command or trigger |
| `job.admin` | Inspect, retry, or cancel durable jobs |
| `memory.read.guild` | Read permitted guild memory scope |
| `relay.dispatch` | Ask Gigi to send a mediated message |
| `relay.receive` | Receive mediated messages from Gigi |

## Rules To Preserve

- Do not grant extra authority because the request came from DM.
- Do not grant extra authority because a plugin created the command.
- Resolve guild, member, channel, role, and target user conservatively.
- Keep plugin permissions auditable by guild and plugin version.
- Keep sensitive actions private or confirmation-gated.
- Fail closed when identity or target resolution is ambiguous.
