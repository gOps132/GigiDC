# Plugin Catalog

In Gigi docs, plugin means an approved external Discord app, bot, or service integration that Gigi can understand through a manifest. It does not mean arbitrary code loaded into Gigi, and Gigi does not run third-party app code.

Plugin catalog work has started. This slice validates, imports, stores, lists, enables, disables, and dry-run matches approved external app manifests, but it does not dispatch external app commands, publish commands declared by manifests, scrape docs, or use LLM routing yet.

## Current Behavior

- external app manifests can be decoded from `gigi-plugin.json`
- manifests must declare exact Discord application ID or bot user ID
- manifests can come from the known catalog, an explicitly provided HTTPS manifest URL, or a Discord attachment upload
- approved manifests can be stored in PostgreSQL
- `/plugins list` shows approved manifests
- `/plugins import-manifest` imports and approves a URL-sourced manifest
- `/plugins import-file` imports and approves an uploaded JSON manifest
- `/plugins enable` enables an approved external app version for the current guild
- `/plugins disable` disables an external app integration for the current guild
- `/plugins enabled` shows enabled guild external app integrations
- enabled guild external app installs can be loaded from PostgreSQL
- guild mentions can dry-run match enabled `prefix` triggers on `guild_text`
- dry-run checks manifest permissions before returning a planned command

v0 does not guess external app identity from fuzzy names, bot display names, domains, or docs pages. Fuzzy suggestions may come later, but exact admin approval remains required.

Future manifest contributions should follow the same runtime format and safety rules in the [contributor guide](/contributing).

## Manifest Requirements

Each manifest must include:

- plugin id, name, version, source, and source kind
- exact Discord application ID or bot user ID
- capabilities
- triggers
- supported surfaces
- required Gigi capabilities
- audit events
- attribution resources

Manifest URL imports must use HTTPS and cannot include user info, query strings, or fragments. Attachment imports must use a `.json` file and are stored as `uploaded_file` source without storing the Discord CDN URL. The current code validates supplied manifest JSON; it does not browse or scrape external app documentation.

```json
{
  "id": "example-tool",
  "name": "Example Tool",
  "version": "1.0.0",
  "source": "builtin",
  "source_kind": "known",
  "discord_application_id": "1511678703963209813",
  "discord_bot_user_id": "1511678703963209814",
  "capabilities": [
    {
      "name": "example.run",
      "description": "Run the external app's declared action."
    }
  ],
  "triggers": [
    {
      "kind": "prefix",
      "value": "!example"
    }
  ],
  "surfaces": ["guild_text"],
  "permissions": ["example.run"],
  "audit_events": ["plugin.example.run"],
  "attribution": [
    {
      "name": "Example Provider",
      "use": "Provide data for the external app action.",
      "source": "https://example.com"
    }
  ]
}
```

## Not Live Yet

- admin-assisted custom manifest creation
- community registry support
- external app command publishing
- actual prefix trigger dispatch to external apps
- LLM tool routing to external app actions
- external app command execution
- docs scraping or browser-based discovery

Next slice should add confirmed dispatch after dry-run matching, permission checks, and audit are proven.
